统一 ChunkServer 底座：对象与文件访问为何可以共用一条数据路径

KyteStore 的核心判断是：对象协议和文件语义在用户入口上不同，但在高性能读写、多副本、本地缓存和冷热分离这些能力上，可以共享同一套数据底座。

从协议差异中抽出共性

湖仓系统更习惯 S3 兼容对象协议，AI 训练和推理任务更依赖 POSIX/FUSE 文件语义。两类入口看起来差异很大，但真正消耗系统资源的是本地 SSD 读写、chunk 分配、数据副本、后台回源和失效回收。KyteStore 把这些共性放在 ChunkServer 中，避免每一种协议都重复实现一套数据路径。

S3 Subsystem 负责对象语义、对象元数据和 S3 API 对接；FS Subsystem 负责目录、inode、extent 和文件可见性。它们之下都使用 ChunkSubSystem 读写本地 chunk，并通过 MetaServer 获取 namespace、slot、owner 和 chunk placement。

ChunkServer 承担什么

  • 统一管理 chunk 生命周期，包括创建、追加、读取、seal、删除和修复。
  • 在本地 SSD 上承载热数据，让高频访问不必每次回源远端对象存储。
  • 提供多副本和 owner fencing，减少单节点故障对业务可见性的影响。
  • 把冷热分离、容量水位和回收策略沉到数据层，使上层协议更轻。

为什么这对用户有价值

用户不需要在对象协议和文件协议之间二选一。湖仓查询可以继续使用 S3 Endpoint，AI 任务可以继续使用文件路径；底层仍然共享本地加速、容量扩展和远端 S3/COS/OSS 回源能力。这种结构也让未来扩展 HDFS、SDK 或其他接入协议时，有明确的复用边界。

这也是 KyteStore 后续性能数据会分为对象加速和文件加速的原因：入口语义不同，benchmark 应该分开观察；但底层的 ChunkServer 能力需要统一设计和统一演进。
