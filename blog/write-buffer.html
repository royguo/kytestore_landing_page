Write Buffer 如何让对象写入先落本地再异步回源

对象存储的远端写入通常受网络、服务端排队和跨区域链路影响。KyteStore 的 Write Buffer 让写入先在本地多副本中完成，再由后台异步同步到远端对象存储。

写入路径

当用户通过 S3 Endpoint 执行 PutObject 或 Multipart Upload 时，请求会进入 S3 Subsystem 的写缓冲命名空间。数据先写入 DataServer 本地 Chunk，并通过 ChunkSubSystem 完成副本同步。对象被 seal 后，新的对象版本立即对后续 GET、HEAD、LIST 可见。

后台同步任务随后把 sealed object 上传到 S3/COS/OSS。对于用户来说，前台写入不再被远端对象存储的单次请求延迟完全绑定；对于平台来说，远端对象存储仍然可以作为容量层和回源层存在。

读后写一致性

Write Buffer 不只是一个异步上传队列。读路径会先检查当前对象是否存在本地写缓冲版本或 tombstone，再决定是否读取本地 chunk、缓存切片或远端对象存储。这样可以避免刚写入的数据被旧的远端版本覆盖。

  • PUT 成功后，GET/HEAD 优先看到本地新版本。
  • DELETE 成功后，读路径识别 tombstone，不再错误回源旧对象。
  • 异步同步成功后，后台 GC 再回收可以安全释放的本地数据。

适合哪些场景

Write Buffer 特别适合 checkpoint、湖仓写入中间结果、批量小对象写入和需要稳定低延迟提交的场景。它不是把远端对象存储替换掉，而是在应用和远端对象存储之间增加一层本地可靠写入面。

评估 Write Buffer 时，除了看 PUT QPS，也要关注本地盘 append 能力、副本数、后台 sync backlog、seal 延迟和失败恢复路径。
